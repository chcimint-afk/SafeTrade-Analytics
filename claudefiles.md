## 🛠️ ЧАСТЬ 1: ТРОЙНОЙ И ШЕСТИКРАТНЫЙ АУДИТ КОДА (INSTAGRAM-МЕТОД)

Каждый раз при изменении кода вы обязаны запустить проверку по всем методам из регламента безопасности. Все консольные команды запускаются строго в директории **`app-code/`**.

1. **ЭТАП 1: ESLint — Анализ логики и чистоты кода**
   * **Где запускать:** `c:/Users/adaml/OneDrive/Bureau/t/app-code`
   * **Команда:** `npm run lint`
   * **Требование:** Ровно `0` ошибок и предупреждений.

2. **ЭТАП 2: NPM Audit — Сканирование внешних библиотек**
   * **Где запускать:** `c:/Users/adaml/OneDrive/Bureau/t/app-code`
   * **Команда:** `npm audit`
   * **Требование:** Уязвимости должны отсутствовать. При необходимости используйте исправления или переопределения (overrides).

3. **ЭТАП 3: TypeScript Compiler (tsc) — Строгая типизация**
   * **Где запускать:** `c:/Users/adaml/OneDrive/Bureau/t/app-code`
   * **Команда:** `npx tsc --noEmit`
   * **Требование:** Полное отсутствие ошибок типизации (`0 errors`). Любые несоответствия типов интерфейсов должны быть исправлены до сохранения.

4. **ЭТАП 4: Secret Scanning — Защита от утечки паролей и токенов**
   * **Инструкция:** ИИ должен проверить весь написанный/измененный код на отсутствие захардкоженных API-ключей, токенов Supabase, паролей или сервисных ролей.
   * **Правило:** Все секретные переменные должны храниться исключительно в файле `app-code/.env.local` или в настройках окружения на Vercel.

5. **ЭТАП 5: Dependency Overrides — Точечное исправление уязвимостей**
   * **Инструкция:** Если `npm audit` находит уязвимости во вложенных зависимостях, принудительно фиксируйте безопасные версии через блок `"overrides"` в `package.json` в папке `app-code/`.

6. **ЭТАП 6: HTTP Security Headers — Сетевой щит терминала**
   * **Инструкция:** ИИ должен проверять, настроены ли заголовки безопасности в `app-code/next.config.ts`.
   * **Обязательные заголовки:** 
     * `X-Frame-Options: DENY` (защита от кликджекинга).
     * `X-Content-Type-Options: nosniff` (защита от MIME-атак).
     * `Referrer-Policy: strict-origin-when-cross-origin`.
     * `Permissions-Policy` (отключение неиспользуемого оборудования вроде камеры или микрофона).

7. **ЭТАП 7: Google Lighthouse — Контроль производительности в продакшене**
   * **Адрес:** `https://safe-trade-analytics.vercel.app`
   * **Инструкция:** Для публичных страниц проекта должен проводиться аудит скорости, SEO и доступности через вкладку Lighthouse в панели разработчика браузера (F12).

8. **ЭТАП 8: CI/CD Pipeline (GitHub Actions) — Автоматический контроль сборки**
   * **Файл конфигурации:** `.github/workflows/ci.yml`
   * **Инструкция:** Любые отправленные в репозиторий изменения автоматически тестируются в облаке GitHub. Пайплайн запускает `npm install`, `npm run lint` и `npx tsc --noEmit`. Модели ИИ обязаны следить за тем, чтобы пайплайн всегда оставался зеленым (успешным).

9. **ЭТАП 9: Automated Testing (Playwright / Jest) — Предотвращение регрессионных багов**
   * **Инструкция:** ИИ-разработчик обязан покрывать критически важную бизнес-логику (расчет рисков, аутентификацию, circuit-breaker) модульными тестами (Jest) и сквозными тестами (E2E с помощью Playwright). Перед деплоем все тесты должны успешно проходить локально.

## ⚡ ЧАСТЬ 2: ШЕСТЬ ИНСТРУМЕНТОВ ИИ (SUPERPOWER & АВТОНОМНЫЕ СУБАГЕНТЫ)

Для исключения ошибок ИИ-разработчик должен использовать все шесть концепций параллельной разработки:

1. **SUPERPOWER (Проектирование и планирование)**
   * **Инструкция:** Не начинайте писать код сразу. ИИ обязан сначала составить файл плана в папке `plan/` и свериться с ТЗ в папке `specs/`. Согласуйте план с пользователем.

2. **FRONTEND DESIGN (Премиальный дизайн)**
   * **Инструкция:** Интерфейсы SafeTrade должны выглядеть как профессиональный терминал Bloomberg. Используйте темные благородные тона, стеклянные эффекты (glassmorphism), плавные микро-анимации при наведении и качественную типографику (шрифты Inter, Roboto).

3. **CODE REVIEW (Параллельные рецензенты)**
   * **Инструкция:** ИИ анализирует код с нескольких сторон одновременно: проверяются синтаксис, логическая чистота, обработка ошибок (блоки `try-catch`) и чистота Git-истории.

4. **SECURITY REVIEW (Контроль доступа)**
   * **Инструкция:** Постоянный аудит авторизации и аутентификации. Защищенные маршруты (например, `/admin/*`) должны строго проверять сессию пользователя на стороне сервера перед рендером контента.

5. **CLAUDE MAP (Векторная память проекта)**
   * **Инструкция:** ИИ обязан изучать карту проекта в папке `docs/` и обновлять её при добавлении новых модулей или изменения архитектуры для поддержания актуального контекста.

6. **G-STACK (Виртуальная IT-компания / Команда субагентов)**
   * **Инструкция:** Для финального тестирования ИИ обязан запустить процесс симуляции независимой команды.
   * **Метод «Спорящих Агентов»:**
     * Создайте виртуального субагента-тестировщика (QA) или аналитика безопасности.
     * Отправьте ему написанный код с заданием: «Найди критические баги, утечки безопасности, слабые места и попытайся сломать эту логику».
     * Агенты должны спорить, указывать на ошибки друг друга, предлагать исправления и приходить к согласию только тогда, когда все проблемы устранены.
     * Код фиксируется и отправляется пользователю только после полного одобрения со стороны субагента-верификатора.
